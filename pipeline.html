Case study · Verify

One dataset, end‑to‑end: how an AI author builds a submission‑grade deliverable that survives independent QC

This follows a single dataset — Demographics — through the whole regulated pipeline of a clinical trial: raw EDC data → SDTM DM → ADaM ADSL → define.xml. An LLM writes each program against an approved specification; an independent QC programmer, architecturally blocked from reading the production code, rebuilds the same dataset from scratch; and a deterministic gate refuses to pass anything until the two builds match byte-for-byte under SAS PROC COMPARE.

The design principle is “validate the wrappers, not the brain.” The model is treated as a qualified author, not an oracle — its output can’t ship until independent, deterministic checks pass and a named human signs off. Below, the gate does exactly its job: it catches a real defect and forces a fix to the specification, not a silent patch to the data.

Study ONC-000-DEV is fully synthetic, calibrated to a published, public trial and unaffiliated with its sponsors. No real patient data appears anywhere.

1Raw EDC data

Collected data as it leaves the electronic data capture system.

The demographics build draws on five CDASH-named EDC extracts from a dated snapshot (raw/snapshots/2026-07-01-crf-driven/) — demographics plus disposition, exposure, and adverse events, because some SDTM variables are derived across domains (death date, actual treatment arm). The primary extract carries the collected fields: 

dm-raw.csv (CDASH columns):
  SUBJID, BRTHDAT, RANDDAT, RFICDAT, ARM, ARMCD, SEX,
  RACE, ETHNIC, COUNTRY, SITEID, RASSTAT, TSFMET, ECOGPS

Collection itself is CRF-driven: the SAP dictates what must be captured, the CRF module specs encode the CDASH fields, and the annotated CRF (aCRF) documents where each SDTM variable comes from. Nothing is programmed until that trace exists.

2SDTM specification

The contract: every variable, its source, and its derivation — written and QC-approved before any code.

The DM specification pins each variable to SDTMIG 3.4, states its exact standard label, and gives an unambiguous derivation. A representative slice of the derived variables:

VariableLabelOriginDerivation
USUBJIDUnique Subject IdentifierAssignedcatx('-', STUDYID, SITEID, SUBJID)
RFSTDTCSubject Reference Start Date/TimeDerivedRANDDAT if ARMCD ≠ NOTASSGN; else null (screen failure)
DTHDTCDate/Time of DeathDerivedEarliest (min) of the DS death date and the fatal-AE date; blank if no death
ACTARMCDActual Arm CodeDerivedRule cascade: not-assigned → not-treated → any bevacizumab ⇒ combination arm → planned arm
AGEAgeDerivedfloor((anchor − imputed_birth)/365.25); anchor = RFSTDTC else RFICDTC

Note the DTHDTC rule: earliest of two sources. That single word is where the QC gate later earns its keep — see stage 4.

3Production SAS program

The AI author writes to the approved spec — and cites it in the header.

Every production program carries machine-checked header tags binding it to its spec, SAP, IG version, controlled-terminology package, and work order. A hook blocks any write that omits them: 

/*-------------------------------------------------------------*
 * Program: dm.sas
 * Purpose: Derive SDTM DM (Demographics) for ONC-000-DEV.
 * @spec    specs/sdtm/DM.md@v3
 * @sap     sap.md#dm
 * @sdtmig  3.4
 * @ct      2026-03-27
 * @wo      WO-2026-0010
 *-------------------------------------------------------------*/

The DTHDTC derivation, implementing the “earliest of two sources” rule verbatim:

/* DTHDTC = earliest (min) of DS death date and fatal AE date; a blank
   source is "no date from that source", not a minimum -- exclude it
   rather than let it win the earliest-date comparison. */
length dthdtc $10;
if      ds_death_dtc = '' and ae_death_dtc = '' then dthdtc = '';
else if ds_death_dtc = ''                       then dthdtc = ae_death_dtc;
else if ae_death_dtc = ''                       then dthdtc = ds_death_dtc;
else if ds_death_dtc <= ae_death_dtc          then dthdtc = ds_death_dtc;
else                                                 dthdtc = ae_death_dtc;

4The QC gate — independent double programming

A second programmer rebuilds the dataset blind; a deterministic compare must find zero differences.

A separate QC programmer derives DM from the same spec in an isolated context, with filesystem-level denial of read access to the production program — the spec is the only channel between them. Then PROC COMPARE runs, and its SYSINFO return code is parsed bit-by-bit. On the first run:

COMPARE_FAIL

65 / 65 obs in common · 0 unequal values

SYSINFO = 32 → bit 5 only: variable labels differ

The data was identical — but the two builds disagreed on variable labels. Root cause: the spec had no authoritative Label column, and the standards lookup script carries no labels, so each programmer chose labels independently. This is a spec ambiguity, not a programmer error — and the gate refused to let it slide. Reconciliation surfaced two further latent defects the independent builds had silently agreed on (a min/max contradiction in the DTHDTC text, and an undercounted actual-arm example).

The fix went to the specification, not the data. DM.md was amended v1 → v3: an authoritative SDTMIG label column added, the DTHDTC rule pinned to earliest (consistent with the SAP’s cutoff-censoring for overall survival), and the actual-arm rule corrected. Both programmers then rebuilt against v3:

COMPARE_OK

65 / 65 obs · 0 unequal values · all attributes (incl. labels) match

SYSINFO = 0 → gate clean

Awaiting a named human sign-off to mark the deliverable validated — the automated gate never grants that status on its own.

5ADaM specification

The analysis-ready layer: population flags and treatment variables, derived from the QC’d SDTM.

ADSL — the subject-level analysis dataset — is built from the now-verified DM (plus EX/DS/AE), per ADaMIG 1.3. The derivations that drive every downstream table:

6ADaM production program

Same gated workflow, one layer up.

ADSL follows the identical pattern as stage 3–4: an AI author programs it against the approved spec above, an isolated QC programmer rebuilds it, and PROC COMPARE must reach SYSINFO = 0 before it ships — with the same @spec / @sap / @adamig header citations. Because ADSL derives almost entirely from the already-verified DM, it inherits a clean, traceable foundation.

Status: the ADSL specification is complete and QC-approved (stage 5); the production program is the next build in this synthetic study. The mechanism is identical to the DM gate already shown — the point of this walkthrough is the governed process, which is proven end-to-end on DM.

7define.xml

Submission-ready, machine-readable metadata — a by-product of the same specs.

Because every variable was specified up front, the regulatory define.xml (Define-XML 2.1) falls out of the pipeline rather than being hand-written afterward. The DM item group, each variable tagged with its origin (Assigned / Collected / Derived): 

<ItemGroupDef OID="IG.DM" Name="DM" Domain="DM" SASDatasetName="DM"
              def:Structure="One record per subject" def:Class="SPECIAL PURPOSE">
  <Description><TranslatedText xml:lang="en">Demographics</TranslatedText></Description>
  <ItemRef ItemOID="IT.DM.USUBJID" OrderNumber="3" Mandatory="Yes" KeySequence="2"/>
  <ItemRef ItemOID="IT.DM.DTHDTC"  OrderNumber="11" Mandatory="No"/>
  ...
</ItemGroupDef>

<ItemDef OID="IT.DM.DTHDTC" Name="DTHDTC" DataType="text" Length="10">
  <Description><TranslatedText xml:lang="en">Date/Time of Death</TranslatedText></Description>
  <def:Origin Type="Derived"/>
</ItemDef>

What this demonstrates: I designed and built this pipeline — the specs, the SAS production and QC programs, the deterministic gate logic, and the standards knowledge layer the citations resolve against. The value isn’t that an LLM writes SAS; it’s that its output is wrapped in independent, deterministic, auditable checks that let it be trusted in a regulated setting. Source is private (it processes confidential trial data and licensed CDISC standards); this walkthrough runs entirely on synthetic ONC-000-DEV.
